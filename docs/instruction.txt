
* Players play many rounds
* Each round:

  * Players are paired
  * Colors (White / Black) are assigned fairly
  * No one plays the same opponent again
  * Nobody gets too many Whites or Blacks
  * One player may get a **bye** if players are odd
* Everything is saved to the database

#  DATA HELPERS (Very Important for DB)

---

## 1️⃣ `serialize_participant_data(participants)`

### What it does (like you’re 10)

Some player data is stored as **lists** in Python
But databases like **strings**

👉 This function:

* Converts lists → JSON strings
* So the database can store them safely

### Example

```python
['white', 'black']  →  "['white','black']"
```

### Where it is called

At the **end of pairing generation**
Before `db.session.commit()`

Node.js equivalent: `JSON.stringify()`

#  COLOR LOGIC (VERY IMPORTANT)

---

## 4️⃣ `get_color_preference(player, round_number)`

### What it does

Decides **what color a player wants**

### Simple explanation

Looks at:

* How many Whites vs Blacks played
* Last games’ colors
* Round number (odd/even)

Then says:

* **ABSOLUTE** → must get this color
* **STRONG** → should get this color
* **MILD** → nice to get this color
* Or no preference

### Returns

```js
{
  type: "absolute" | "strong" | "mild",
  color: "white" | "black",
  games_played: number,
  mild_adjustable: true/false
}
```

### Where used

Used by:

* `would_violate_color_rules`
* `can_pair`
* `calculate_pairing_quality`
* `assign_colors`

---

## 5️⃣ `would_violate_color_rules(player, assigned_color)`

### What it does

Checks:
❌ Will this color break rules?

Rules:

1. No 3 same colors in a row
2. Don’t break **absolute preference**

### Returns

* `True` → ❌ not allowed
* `False` → ✅ allowed

### Where used

* `can_pair`
* `assign_colors`
* `calculate_pairing_quality`

---

## 6️⃣ `can_pair(p1, p2)`

### What it does

Checks if two players **can legally play**

Checks:

* Haven’t played before
* At least one valid color assignment exists

### Where used

* Inside pairing logic
* Before trying to match players

---

## 7️⃣ `colors_are_compatible(p1, p2)`

### What it does

Quick check:

* Do players want **opposite colors**?

If yes → good pairing

### Where used

Used as a **fast filter** during pairing attempts

---

## 8️⃣ `calculate_pairing_quality(p1, p2)`

### What it does

Gives a **score number**

* Lower number = better pairing

### Penalizes:

* Big score difference
* Same color preference
* Bad float history

### Used for

👉 Choosing **best possible pairing** among many options

### Where used

* `pair_bracket_with_color_priority`

---

## 9️⃣ `assign_colors(p1, p2, round_number)`

### What it does

 **Decides who gets White and Black**

Uses priority order:

1. Absolute preferences
2. Strong preferences
3. Mild preferences
4. Higher-ranked player
5. Balance colors

### Returns

```python
(white_player, black_player)
```

### Where used

* Final pairing creation
* Round 1 pairing
* Later rounds

#  BYE HANDLING

---

## 🔟 `select_bye_player(players)`

### What it does

Chooses who gets a **bye**

Rules:

* Lowest score
* Fewest byes so far
* Tie → higher ID loses (fairness)

### Where used

* When player count is odd

#  SCORE BRACKET PAIRING

---

## 1️⃣2️⃣ `pair_bracket_with_color_priority(players)`

### What it does

Pairs players **inside same score group**

Steps:

1. Try top-half vs bottom-half
2. Respect color rules
3. Pick best-quality pairing
4. Leave leftovers as **floaters**

### Returns

```python
(pairs, floaters)
```

### Where used

Main Swiss pairing for rounds ≥ 2
